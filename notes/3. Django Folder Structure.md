# Django Folder Structure Explained

## Introduction

When you create a new Django project, Django automatically generates a folder structure. Understanding this structure is very important because it helps you know where to write code and how Django works internally.

This document explains each folder and file in simple language.

---

## Basic Project Structure

After running:

```bash
django-admin startproject mysite
```

You get:

```
mysite/
├── manage.py
└── mysite/
    ├── __init__.py
    ├── settings.py
    ├── urls.py
    ├── asgi.py
    └── wsgi.py
```

---

## 1. manage.py

Location:

```
mysite/manage.py
```

Purpose:

* Main command-line tool for the project
* Used to run server, migrate database, create users, etc.

Examples:

```bash
python manage.py runserver
python manage.py migrate
python manage.py createsuperuser
```

It connects your project with Django’s internal system.

---

## 2. Project Folder (Inner mysite/)

Location:

```
mysite/mysite/
```

This folder contains core project configuration files.

It is the brain of your Django project.

---

## 3. **init**.py

Purpose:

* Makes this folder a Python package
* Allows Python to import files from this folder

Usually left empty.

---

## 4. settings.py

Purpose:

* Main configuration file
* Controls how your project works

Contains:

* Database settings
* Installed apps
* Middleware
* Templates
* Static files
* Security settings

Example:

```python
INSTALLED_APPS = []
DATABASES = {}
```

Most important file in the project.

---

## 5. urls.py

Purpose:

* Handles URL routing
* Connects URLs to views

Example:

```python
path('', home)
```

It decides what happens when a user visits a URL.

---

## 6. asgi.py

Purpose:

* Used for asynchronous servers
* Supports WebSockets and real-time features
* Used in modern deployments

ASGI = Asynchronous Server Gateway Interface

---

## 7. wsgi.py

Purpose:

* Used for traditional web servers
* Helps deploy Django projects

WSGI = Web Server Gateway Interface

---

## App Folder Structure

After running:

```bash
python manage.py startapp core
```

You get:

```
core/
├── admin.py
├── apps.py
├── models.py
├── tests.py
├── views.py
└── migrations/
```

---

## 8. admin.py

Purpose:

* Registers models to Django Admin Panel
* Allows easy data management

---

## 9. apps.py

Purpose:

* Stores app configuration
* Defines app name

Example:

```python
class CoreConfig(AppConfig):
```

---

## 10. models.py

Purpose:

* Defines database tables
* Written using Python classes

Example:

```python
class User(models.Model):
```

---

## 11. views.py

Purpose:

* Contains business logic
* Handles requests and responses

Example:

```python
def home(request):
```

---

## 12. tests.py

Purpose:

* Used for writing test cases
* Helps check app correctness

---

## 13. migrations/

Purpose:

* Stores database changes
* Keeps track of model updates

Django automatically manages this folder.

---

## Common Extra Folders (Advanced Projects)

In real projects, you may also see:

```
templates/
static/
media/
.env
requirements.txt
```

### templates/

Stores HTML files

### static/

Stores CSS, JS, images

### media/

Stores uploaded files

### .env

Stores secret keys

### requirements.txt

Stores project dependencies

---

## How Everything Works Together

Request Flow:

```
Browser → urls.py → views.py → models.py → template → Response
```

Each folder plays a role in this flow.

---

## Best Practices

* Do not modify core Django files
* Keep apps small and focused
* Organize templates properly
* Use virtual environment
* Commit clean structure to GitHub

---

## Memory Trick

Think of Django project like a company:

* settings.py → Rules
* urls.py → Reception
* views.py → Workers
* models.py → Database
* templates → Design Team
* admin.py → Manager

---

## Conclusion

Django folder structure is designed to keep projects clean and scalable.

Understanding it helps you:

* Write better code
* Debug faster
* Work in teams
* Build large projects
